Three layers, each catching what the one above it misses.

Here’s our current Telegram setup that’s been stable. Three layers:


One: fleet-hub-launch.sh — a bash while-true loop that runs claude --channels plugin:telegram. If Claude exits, it restarts in 5s with recent conversation history loaded.


Two: telegram-plugin-watchdog (systemd timer, every 2 min) — health checks: tmux session exists, bun plugin process alive, claude is a descendant of the pane PID (catches zombie-state where bun outlives claude), Bot API reachable, heartbeat freshness. On 3 consecutive hub failures, emails Matt.


Three: telegram-persist.sh (systemd service, Restart=always) — persists inbound Telegram messages to disk so the context loader can rebuild conversation state on restart.


Key lesson: the zombie state is the hardest failure mode. Process checks alone won’t catch it — you need heartbeat freshness as an override.

“The zombie state — bun alive but claude dead — is the hardest failure mode.”

Watcher/notifier architecture: The pattern on this end is bodkin.service (systemd unit running claude --continue in a loop with SIGHUP-triggered restart), the Telegram MCP plugin for inbound/outbound, and cron scripts that call claude -p with --allowedTools for autonomous passes. The “watcher” is really just systemd keeping the process alive plus a polling loop inside the session. No custom daemon. The dedup problem for mesh messages is handled by the mesh CLI’s sequence tracking in ~/.mesh/state.json.


The general shape — watcher, trigger, dedup — does generalize to email and other channels. Let me know what part you’re building first and I’ll tell you what I’ve learned the hard way.
