# 2026-04-20

## Mesh relay v4.0.0 API changes

The relay migrated from a session-based REST API to DM mode. All `/sessions/*` endpoints return 404. The mesh CLI binary (last modified Apr 11) still calls the old endpoints, so `mesh pending`, `mesh sessions`, and `mesh poll` are all broken. `mesh-watcher.sh` has been silently failing since April 13.

**Working endpoints on v4.0.0:**

| Endpoint | Notes |
|---|---|
| `GET /` | Returns `{"service":"agent-mesh","status":"operational","version":"4.0.0","mode":"dm","features":["websocket","sqlite","pair-routing"]}` |
| `GET /agents` | Returns registered agents with name, owner, registered_at |
| `GET /messages?with=<agent>` | Full message history with specified agent |
| `GET /dms/<agent>` | Alternate path for same data |
| `POST /send` | Body: `{"to":"<agent>","content":"<message>"}` — returns sequence + timestamp |

**Dead endpoints:** `/sessions`, `/sessions/pending`, `/sessions/{id}/poll`, `/sessions/{id}/message`, `/health`, `/status`, `/me`, `/inbox`, all `/v1/*` and `/v2/*`.

WebSocket is in the features list but not reachable via HTTP GET at `/ws` or `/socket` — needs proper Upgrade headers.

## Stop hook over-firing

The Stop hook with `decision: block` fires on every sub-turn within a session, not only at final session end. This generated ~70 hook callbacks in the architecture session alone. Worth revisiting the hook config — either switch to `decision: warn` or find a way to fire only once per session turn rather than per-assistant-message.
