# Client writing outline

## Purpose

This skill produces a structured, client-facing content outline using Roman numeral format. The outline serves as a reviewable document — something a client or project manager can respond to before full drafting begins. It is not an internal planning document. Everything in it, including comments, should be written as if the client will read it.

## Outline format

Use nested Roman numeral format:
- Level I: Top-level sections (e.g., Page 1, Page 2, or named document sections)
- Level A: Subsections within each top-level section
- Level 1: Content elements within each subsection
- Level a: Copy or descriptive notes within each content element
- Level i: Sub-points when needed (use sparingly)

Single-spaced throughout. No extra line breaks between items at the same level.

## What to include at each level

**Top-level sections (Level I):** Named for their structural role in the deliverable — e.g., "Page 1," "Page 2," or named sections of a document.

**Subsections (Level A):** Named for what the reader will see on the page — the display name, not an internal label. For example, "Chat with your data" not "Use case 1." Where an internal organizational label is needed, note it explicitly with a bracketed clarifier:

*["Use case 1" is an internal label only — "Chat with your data" is the proposed on-page display name, open to refinement.]*

**Content elements (Level 1):** Named for their structural role within the subsection. Use plain, universally understood terms:
- "Headline" not "H1"
- "Intro line" or "subtitle" not "subhead"
- "Closing line" not "closer" or "inclusive closer"
- "Opening stat" for a standalone callout number
- "Business problem," "What changed," "Business outcome," "Closing line" are acceptable working labels — but note explicitly that these are for review purposes, not intended as on-page headers

**Copy (Level a):** Actual drafted copy, written as it would appear on the page. Even in a high-level outline, copy should be as close to final as available information allows. Do not write placeholder copy like "TBD — insert copy here." If a section can't be fully drafted, write the best available version and use a comment to flag what's missing.

**Visual descriptors:** Where layout or visual treatment matters for the client's understanding, add a bracketed descriptor inline with the element label:
- `Headline [large display text]`
- `Intro line/subtitle [smaller text just beneath headline]`
- `Opening stat [standalone callout — large number with short label, positioned alongside or just below the intro line]`

These descriptors communicate layout intent without requiring a design mockup.

## Comments

Comments are the primary vehicle for flagging open questions, decisions pending client input, and anything that needs to be resolved before drafting. Keep comments in the outline rather than in a separate document — they surface the ask in context.

**Comment style:**
- Plain, conversational language — no insider terminology
- No "we" — use passive voice or impersonal constructions ("confirmation needed," "flagged for review")
- No second person ("you") directed at the client
- No jargon: not "outcome beat," "four-beat treatment," "framework," "content module"
- State what the issue is, why it matters, and what action is needed — in that order
- One short paragraph at most per comment

**What comments are for:**
- Flagging open information needs (missing customer story, stat approval pending, CTA direction unconfirmed)
- Explaining a content decision the client hasn't yet confirmed
- Noting where copy would be stronger with additional input
- Distinguishing internal working labels from proposed on-page display copy

**What comments are not for:**
- Explaining the writing process or methodology to the client
- Summarizing decisions already made
- Anything the client doesn't need to act on

## Language principles for all outline copy

Apply these to copy and comments throughout:

- **Businesses as protagonist:** Businesses (not AI, not technology) should be the grammatical subject wherever possible. "Businesses are using AI to save time" not "AI is helping businesses save time."
- **Avoid noun phrase repetition:** If "growing businesses" appears in the headline, use "businesses of every size" or a similar variation in body copy.
- **Sentence case** for all headings, labels, and copy — never title case.
- **No contrastive negation:** Avoid "it's not X, it's Y" constructions.
- **No line dividers** in the document.
- **Plain language:** Toned-down, not polished or flowery. Short sentences. Outcomes and specifics up front; setup and context after.
- **No collective "we":** First-person singular or passive voice only.

## Handling incomplete information

When information needed for a section hasn't arrived yet (customer story, stat approval, CTA direction):

1. Write the best available version of the copy using general language or directional framing grounded in available source material
2. Add a comment flagging what's missing, why it matters, and what the client needs to provide

This keeps the outline moving and gives the client something concrete to react to.

## Working labels vs. on-page display names

Always distinguish between labels used for internal organization and copy that will appear on the page:

- Section numbers ("Use case 1") are internal — never appear on the page
- Display names ("Chat with your data") are proposed on-page copy — explicitly note they're open to client input
- Content structure labels ("Business problem," "What changed") are working labels for review — note in a comment that these are not proposed on-page headers

## Tone check before sharing

Read through the outline as if seeing it for the first time as the client. Ask:
- Does any label or comment use insider content strategy terminology?
- Is any sentence longer than it needs to be?
- Does any copy make AI or technology the protagonist instead of the business?
- Does any comment use "we" or address the client as "you"?
- Does any copy sound polished in a way that calls attention to itself?

Revise anything that fails this check before the outline is shared.
